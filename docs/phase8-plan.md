# Phase 8: Reduce PIE Binary SRAM Footprint

**PicoPiAndPortable Development Roadmap**
Estimated Duration: 2–3 days
Prerequisites: Phase 6 (musl + busybox) complete

---

## Context

Post-boot free memory is approximately **60 KB**.  Two resident processes
(init + sh) each load the full BusyBox binary, whose data segment consumes
~34 KB per instance.  Two optimisations target this:

**Part A — .rodata flash migration** (~25 KB saved per process)

Analysis of the BusyBox PIE binary reveals that its data segment (copied to
SRAM by `exec.c`) is ~33.8 KB, of which ~25 KB is `.rodata` — read-only
constant data that does not *need* to be writable.  The `.rodata` section is
placed in the RW data segment solely because the linker script conservatively
assumes any `.rodata` entry might contain pointers requiring `R_ARM_RELATIVE`
relocation.

Actual relocation profile (BusyBox 1.1 MB PIE binary):

| Section  | Size     | R_ARM_RELATIVE entries | Needs SRAM? |
|----------|----------|----------------------|-------------|
| `.got`   | 3 136 B  | 777                  | Yes (GOT)   |
| `.rodata`| 25 421 B | **16** (64 bytes)    | Only 64 B   |
| `.data`  | 953 B    | 111                  | Yes          |
| `.bss`   | 4 260 B  | 0                    | Yes          |

The 16 relocations inside `.rodata` belong to a single unnamed jump table
generated by libgcc's soft-float `__aeabi_ddiv`.  Every other `.rodata` entry
(locale tables, error strings, applet dispatch tables, parser rules, etc.) is
pure constant data with no embedded pointers.

**Part B — Split init/sh into dedicated binaries** (~12 KB saved for residents)

After Part A, each BusyBox instance still allocates 3 SRAM pages (12 KB) for
its data segment (.got + .data + .bss).  The GOT alone is 3 136 B (784
entries) because it references symbols from *all 28 applets*.  A purpose-built
init or sh binary links only the functions it actually needs, dramatically
shrinking the GOT and reducing page allocation:

| Binary          | GOT est.   | Data seg est. | Pages |
|-----------------|-----------|---------------|-------|
| Full BusyBox    | 784 (3 136 B) | ~8.5 KB   | 3 (12 KB) |
| init-only       | ~80 (320 B)   | ~3–4 KB   | 1 (4 KB)  |
| sh-only (hush)  | ~260 (1 040 B)| ~5–6 KB   | 2 (8 KB)  |

Resident process SRAM: 6 pages (24 KB) → 3 pages (12 KB).  **Saves 12 KB
(3 pages).**

**Combined goal:** reclaim ~37 KB of SRAM (~25 KB per-process rodata +
12 KB resident page reduction), increasing post-boot free memory from ~60 KB
to ~85+ KB (≈ 40 % improvement).

## Part A — .rodata Flash Migration

### Approach

Use GCC's `-fdata-sections` to split per-symbol `.rodata` into named
subsections (`.rodata.<name>`), then update the linker script to place named
subsections in the **text segment** (flash XIP) while keeping bare `.rodata`
(from pre-compiled libgcc, which does *not* use `-fdata-sections`) in the
**data segment** (SRAM).

This is a build-time–only change: **no kernel code changes** required.

Key insight:
- With `-fdata-sections`, musl and BusyBox `.o` files emit per-variable
  sections: `.rodata.table.0`, `.rodata.errmsgstr`, etc.
- Pre-compiled libgcc.a was built *without* `-fdata-sections`, so its rodata
  stays in the bare `.rodata` section.
- The 16-entry `__aeabi_ddiv` jump table (the only rodata needing relocation)
  is from libgcc → remains in bare `.rodata` → stays in the SRAM data segment.
- All other rodata (~25 KB) moves to flash.

### Linker script section routing (after change)

```
Text segment (flash XIP, PF_R|PF_X):
  .rodata.str  ←  *(.rodata.str*)       # string pools (already here)
  .rodata.cst  ←  *(.rodata.cst*)       # numeric constants (already here)
  .rodata.sub  ←  *(.rodata.*)          # NEW: named subsections → flash

Data segment (SRAM, PF_R|PF_W):
  .got         ←  *(.got*)
  .rodata      ←  *(.rodata)            # CHANGED: bare .rodata only (libgcc)
  .data        ←  *(.data*) *(.data.rel.ro*)
  .bss         ←  *(.bss*) *(COMMON)
```

GNU LD processes input sections in declaration order; each input section is
placed only once.  `.rodata.str*` and `.rodata.cst*` are consumed first, then
`*(.rodata.*)` captures remaining named subsections, and `*(.rodata)` catches
only the bare section.


### Step 3 — Update linker scripts

**`third_party/configs/busybox.ld`:**

In the text segment, after `.rodata.cst`, add a new output section to capture
named `.rodata.*` subsections:

```ld
/* Named .rodata subsections (from -fdata-sections).
 * Pure constants — no embedded pointers.  Safe in flash. */
.rodata.sub : {
    *(.rodata.*)
} :text
```

In the data segment, change the existing `.rodata` rule from
`*(.rodata*)` (catch-all) to `*(.rodata)` (bare section only):

```ld
/* Bare .rodata (pre-compiled libgcc without -fdata-sections).
 * May contain jump tables with R_ARM_RELATIVE entries. */
.rodata : {
    *(.rodata)
} :data
```

**`user/user.ld`:** apply the same pattern for consistency (user programs are
small, but this prevents surprises if they grow).

Then check the new segment layout:


All 16 `.rodata` relocations should remain inside the data segment (bare
`.rodata`), plus the existing `.got` and `.data` relocations.

## Part B — Split init/sh Into Dedicated Binaries

### Approach

Build three BusyBox variants from the same source tree, each with a different
`.config`:

| Variant | Applets | Role | Flash (text) |
|---------|---------|------|-------------|
| `busybox` | All 28 applets | Transient commands (ls, cat, grep, ...) | ~160 KB |
| `busybox.init` | `CONFIG_INIT` only | PID 1 (resident) | ~30 KB |
| `busybox.sh` | `CONFIG_HUSH` + builtins only | Interactive shell (resident) | ~70 KB |

Each variant shares the same musl libc, libgcc, and linker script, but links
only the applet code it needs.  The linker's `--gc-sections` (already implied
by `-fdata-sections` + `-ffunction-sections`) strips unreachable code.

Init spawns `/bin/sh` (the dedicated shell binary) instead of using BusyBox
`FEATURE_SH_STANDALONE`.  When `sh` runs a command (e.g. `ls`), it still
`execve("/bin/busybox", ...)` — the full binary, loaded transiently.

### Why this saves SRAM

The GOT is the dominant SRAM consumer after Part A.  Each unique symbol
reference adds a 4-byte GOT entry.  Full BusyBox references symbols from
28 applets + hush + init → 784 GOT entries (3 136 B).  A dedicated init only
references ~80 symbols → ~320 B GOT.  Combined with smaller `.data` and
`.bss` (no applet dispatch tables, no shell parser state), the data segment
fits in fewer 4 KB pages.

## Risks and Mitigations

| Risk | Likelihood | Mitigation |
|------|-----------|------------|
| Some musl/busybox rodata stays in bare `.rodata` | Low | Step 1 gate verifies; if found, use objcopy `--rename-section` |
| `-fdata-sections` increases flash binary size | Low (~1–2 KB) | Acceptable trade-off for 25 KB SRAM saving |
| libgcc has more bare `.rodata` than just 64 B | Low | Verify with readelf; even a few hundred bytes is fine |
| Named `.rodata` subsection contains pointers | Very low | Re-run relocation analysis after rebuild |
| init-only busybox fails to start shell | Low | Verify inittab and execve path in Step 11 |
| `FEATURE_SH_STANDALONE` breaks with split binaries | Medium | Disable it in sh fragment; shell always forks + execs for non-builtins |
| Flash budget exceeded (2 MB) | Very low | 288 KB is 14 % of 2 MB; plenty of headroom |

## Fallback: exec.c Shadow Relocation

If a future binary has `.rodata` relocations that cannot be separated at build
time (e.g., a library compiled without `-fdata-sections` that mixes pointers
and constants in the same section), the kernel loader can be enhanced:

1. Move all `.rodata` to text segment in the linker script.
2. In `exec.c`, after GOT relocation, scan `.rel.dyn` for `R_ARM_RELATIVE`
   entries targeting the text segment.
3. For each cluster of text-segment relocations:
   - Allocate a small SRAM "shadow buffer" (e.g., 64 B).
   - Copy the affected region from flash, patch relocations in the copy.
   - Scan the GOT for entries pointing into the flash range of the cluster;
     redirect them to the SRAM shadow.
4. Net cost: ~50 lines in `exec.c`, ~64 B SRAM per binary with text-segment
   relocations.

This fallback is **not required for the current BusyBox** but provides a
general-purpose safety net.

## Design Note: Why Not Split All Applets Into Individual Binaries?

We considered building every applet (ls, cat, grep, …) as its own binary
instead of keeping a single full BusyBox for transient commands.  Analysis
showed this is not worthwhile.

### The musl duplication problem

Each PIE binary statically links its own copy of musl libc and libgcc.  Common
runtime functions (malloc, stdio, printf, soft-float arithmetic, string ops,
syscall wrappers) total ~34 KB of text per binary.  With 28 individual applet
binaries, this shared code is duplicated 28 times:

| Configuration | romfs text (stripped) | Flash overhead |
|---------------|---------------------|----------------|
| Single full BusyBox | ~193 KB | baseline |
| + init + sh split (Part B) | ~288 KB | +95 KB |
| All 28 applets individual | ~1 050 KB | **+857 KB** |

The all-individual approach consumes ~1 MB of the 2 MB flash — over half —
just for musl/libgcc duplication.

### Marginal SRAM benefit

Splitting init and sh (Part B) saves 12 KB of **permanently resident** SRAM.
Splitting the remaining 26 transient applets saves 4–8 KB **per concurrent
process**, but only while that process is running (typically tens of
milliseconds for simple commands).

The only scenario where transient savings matter is concurrent pipelines:

```
cat file | grep pattern | sort      # 3 concurrent busybox instances
```

| Configuration | Pipeline SRAM (3 stages) |
|---------------|-------------------------|
| Full BusyBox (Part A) | 3 × 12 KB = 36 KB |
| Individual binaries | 3 × 4–8 KB = 12–24 KB |
| Saving | 12–24 KB (temporary) |

This is meaningful but transient — the pages are freed as soon as the pipeline
completes.

### Conclusion

| | Part B (init+sh only) | All individual |
|--|----------------------|----------------|
| Resident SRAM saved | **12 KB** | 12 KB (same) |
| Transient SRAM saved | 0 | 12–24 KB (temporary) |
| Flash cost | +95 KB | **+857 KB** |
| Build complexity | 3 configs | 28+ configs |

Part B captures the permanent savings at modest flash cost.  All-individual
provides only transient savings at 9× the flash cost.  If pipeline memory
pressure becomes an issue in practice, a targeted split of frequently piped
applets (grep, sed, sort) can be revisited.

## Part C — Build Hygiene and Per-Target Configuration

Post Part A+B clean-up: gate test code, init config fixes, conditional
SD/VFAT/UFS compilation, and per-target romfs generation.

### Changes

1. **PPAP_TESTS gating** — kernel self-tests (kmem, PROC, XIP verify) now
   compile only when `PPAP_TESTS=1`.  User-space test binaries (test_vfork,
   runtests, etc.) only built/installed when the flag is set.  Stale test
   binaries removed from romfs/bin on non-test builds.

2. **flash.sh rewrite** — accepts target name (pico1, pico1calc, qemu_arm)
   instead of ELF path.  `--test` flag passes `-DPPAP_TESTS=ON` to cmake.
   `--build` flag skips flash (build only).

3. **busybox.init: FEATURE_INIT_SYSLOG** — added to init fragment to suppress
   `/dev/tty5` log errors (tty5 doesn't exist on PPAP).

4. **PPAP_HAS_BLKDEV compile flag** — gates SD card, VFAT, UFS, block device,
   and loopback support at build time:

   | Target | PPAP_HAS_BLKDEV | Effect |
   |--------|-----------------|--------|
   | pico1 | OFF | No blkdev/vfat/ufs code; ~20 KB flash saved |
   | pico1calc | ON | Full SD + VFAT + UFS + loopback |
   | qemu_arm | ON | ramblk + VFAT + UFS + loopback |

   Files gated with `#ifdef PPAP_HAS_BLKDEV`:
   - `main.c` — blkdev_init/loopback_init calls
   - `fstab.c` — vfat/ufs mount paths
   - `procfs.c` — /proc/mounts vfat/ufs entries
   - `devfs.c` — /dev/mmcblk0, /dev/loopN devices
   - `sys_fs.c` — sys_mount vfat/ufs support

   CMake: `KERNEL_BLKDEV_SOURCES` list separated from `KERNEL_COMMON_SOURCES`;
   only linked into pico1calc and qemu_arm targets.

5. **Per-target romfs** — each target gets its own romfs.bin with an
   appropriate `/etc/fstab`.  Base fstab has pseudo-FS only (devfs, procfs,
   tmpfs).  PPAP_HAS_BLKDEV targets get `mmcblk0 /mnt/sd vfat rw` appended.
   Staging directories (`build/romfs_<target>/`) ensure parallel builds work.
